# Метрики

## Зачем они в Search Lab

Метрики позволяют **сравнивать алгоритмы** на одних и тех же запросах и разметке: насколько хорошо ранжируются документы, насколько быстро отвечает поиск, насколько «отделяются» оценки релевантных и нерелевантных результатов.

Есть два разных слоя:

1. **Прикладные метрики** — считаются в сервисе; работают для **bm25**, **semantic**, **hybrid**, **rrf**.
2. **Нативный OpenSearch `_rank_eval`** — только **BM25**, фиксированный набор имён метрик; **NDCG там недоступен**.

Для перечисленных ниже метрик качества (кроме задержки) **больше — лучше**.

## Прикладные метрики (`POST /api/v1/eval` и бенчмарк)

Вы задаёте список **релевантных ID** и сравниваете его с выдачей.

### NDCG@K (нормализованный дисконтированный кумулятивный выигрыш)

**Смысл:** поощряет расположение релевантных документов **выше** в списке; вклад позиций ниже по рангу **уменьшается** (дисконтирование).

**Диапазон:** обычно **0–1** (нормализованная величина).

**Интерпретация:** **1** — идеально относительно вашей разметки; **0** — нет полезного сигнала в верхней части выдачи.

### MRR (средний обратный ранг)

**Смысл:** для одного запроса берётся **1 / ранг первого релевантного** документа (например, первый релевантный на 3-м месте → 1/3). В бенчмарке усредняется по шаблонам.

**Диапазон:** **0–1**.

**Интерпретация:** **1** — первый результат всегда релевантен; меньшие значения — пользователю чаще нужно листать дальше.

### Precision@K (точность на K)

**Смысл:** доля **релевантных** среди **первых K** возвращённых документов.

**Диапазон:** **0–1**.

**Интерпретация:** насколько «плотная» первая страница хороших результатов.

### Recall@K (полнота на K)

**Смысл:** какую долю **всех** отмеченных вами релевантных ID удалось показать **в пределах топ-K**.

**Диапазон:** **0–1**.

**Интерпретация:** насколько вы «нашли» известные хорошие документы в одной выдаче. При одной метке релевантности часто получается 0 или 1.

## Метрики только в бенчмарке

### `latency_ms`

**Смысл:** время выполнения поиска для пары (алгоритм × шаблон), в миллисекундах.

**Интерпретация:** **меньше — лучше** для отзывчивости; сравнивайте на сопоставимом железе и данных.

### `score_separation`

**Смысл:** средняя оценка **релевантных** попавших в выдачу минус средняя оценка **нерелевантных** в той же выдаче.

**Интерпретация:** **большее положительное значение** — ранжировщик в среднем ставит релевантные документы **выше по скору**, чем прочие в этом же списке; полезно, когда NDCG/MRR близки.

## Нативный `_rank_eval` (`POST /api/v1/eval/rank-eval`)

Вызов встроенного API OpenSearch для **BM25** на **одном логическом индексе**. Не используйте ключ **`all`**: разметка релевантности относится к **одному** физическому индексу.

**Допустимые значения поля `metric`:**

| Значение | Назначение |
|----------|------------|
| `dcg` | Discounted Cumulative Gain (часто значение по умолчанию в API) |
| `precision` | Метрика в духе точности на K |
| `recall` | Метрика в духе полноты на K |
| `mean_reciprocal_rank` | Метрика в духе MRR |

**NDCG в этом API не поддерживается.** Для **NDCG@K** используйте **`POST /api/v1/eval`** или **бенчмарк**, где NDCG считается на стороне приложения.

## Где смотреть в UI

- **Поиск** — панели **Eval marks** (прикладные метрики для одного запроса) и **rank-eval** (нативный BM25, один индекс).
- **Эксперименты → Бенчмарк** — матрица результатов: NDCG@K, MRR, Precision@K, Recall@K, **latency_ms**, **score_separation** и сводки по алгоритмам.
